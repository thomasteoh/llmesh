OpenAI-compatible endpoint
Drop-in replacement for api.openai.com
POST /v1/chat/completions
Authentication
Authorization: Bearer sk-yourkey
Example (streaming)
curl https://{{.Host}}/v1/chat/completions \ -H "Authorization: Bearer sk-yourkey" \ -H "Content-Type: application/json" \ -d '{"model":"llama3.2:3b","messages":[{"role":"user","content":"hi"}],"stream":true}'
Python SDK
from openai import OpenAI client = OpenAI(base_url="https://{{.Host}}/v1", api_key="sk-yourkey")
Model not found — error response

If the requested model or alias is not served by any connected client, the router returns HTTP 400 with a list of what is currently available:

{"error":{"message":"model \"xyz\" not found","available_models":["llama3.2:3b"],"available_aliases":{"small":"llama3.2:3b"}}}

Shim config reference
All fields for shim-config.yaml

llmesh-shim connects to the router using the same WebSocket protocol as llmesh-client, but dispatches jobs to external HTTP APIs or shell command adapters instead of a local llama.cpp server. No GPU required.

Full example
router_url: "wss://{{.Host}}/ws/client" router_token: "ct-yourname-yourtoken" max_concurrent: 4 models: - name: "gpt-4o" context_size: 128000 backend: type: http url: "https://api.openai.com" format: openai auth_type: bearer auth_value: "${OPENAI_API_KEY}" - name: "claude-sonnet-4-5" context_size: 200000 backend: type: http url: "https://api.anthropic.com" format: anthropic auth_type: bearer auth_value: "${ANTHROPIC_API_KEY}" # Shell command adapter - name: "my-model" backend: type: command command: "python3 /path/to/adapter.py"

commandyesShell command (passed to sh -c). Receives JSON-encoded InferenceRequest on stdin. Supports ${VAR} expansion.
Command adapter protocol

The adapter reads a JSON InferenceRequest from stdin and writes to stdout.

Batch (stream: false) — one JSON line:

{"content":"response text","finish_reason":"stop"}

Streaming (stream: true) — NDJSON, one line per chunk:

{"delta":"Hello","done":false} {"delta":" world","done":false} {"delta":"","done":true,"finish_reason":"stop","usage":{"prompt_tokens":10,"completion_tokens":5,"total_tokens":15}}
Notes

${VAR} references in auth_value and command are expanded from the environment at startup. Pass API keys via environment variables rather than hardcoding them in the config file.

The shim sends an empty keep-alive chunk every 60 seconds during long upstream calls to prevent the router’s activity timeout from expiring.

high → processed first (0) normal → default (1) low → processed last (2)

Set priority per API key on the API Keys page. Admins can update priority at any time.

Affinity scheduling: when a client is free it prefers to pick up its owner’s requests first. Affinity overrides priority tier — an owner’s normal-priority request beats another user’s high-priority request for that client. If the owner has no queued requests, any eligible request is used as fallback.

Model aliases
Server-managed short names that route to any available matching model

Aliases are managed inline on the Clients page, not in client config files. Clients register their full model names only. The router resolves aliases at dispatch time.

Example

Shared aliases

Multiple clients can serve models with the same alias. A caller requesting "qwen" will be routed to whichever matching client has spare capacity first — load-balancing and affinity rules apply as normal. The alias is resolved to the canonical model name at dispatch time.

Tool calling
Function/tool calls passed through to llama.cpp

Tool definitions, tool calls, and tool results are passed through the full request pipeline without modification. The router preserves the raw JSON structure of tools, tool_calls, and tool_result content blocks at every stage.

OpenAI format
curl https://{{.Host}}/v1/chat/completions \ -H "Authorization: Bearer sk-yourkey" \ -d '{ "model": "llama3.2:3b", "tools": [{"type":"function","function":{"name":"get_weather","parameters":{...}}}], "messages": [{"role":"user","content":"What is the weather in Paris?"}] }'
Anthropic format
curl https://{{.Host}}/v1/messages \ -H "x-api-key: sk-yourkey" \ -d '{ "model": "llama3.2:3b", "tools": [{"name":"get_weather","description":"...","input_schema":{...}}], "messages": [{"role":"user","content":"What is the weather in Paris?"}] }'

Multi-turn tool use

Conversations with tool_use / tool_result content blocks (Anthropic) or tool_calls / tool role messages (OpenAI) are forwarded intact. The full message history including tool call results must be included in each request — the router is stateless.

Note on format translation

llama.cpp speaks OpenAI format. When using the Anthropic endpoint with tools via llmesh-client, the model must support OpenAI-style tool calling. Anthropic↔OpenAI tool format translation is not performed by the router.

When using llmesh-shim, tools are forwarded in the native format of the configured backend (OpenAI or Anthropic), so Anthropic-format tool blocks reach Anthropic APIs unmodified.

{{/* ── Chat templates ─────────────────────────────────────── */}}
Chat templates
Override the model’s built-in Jinja2 prompt format

llama.cpp applies a Jinja2 chat template to format the message list into a prompt before feeding it to the model. By default it uses the template embedded in the GGUF file. If the embedded template is wrong, missing, or you want to experiment with a different format, set chat_template per model in config.yaml.

Example
models: - name: "unsloth/qwen3-30b-a3b" endpoint: "http://localhost:8081" chat_template: "qwen2.5" # use llama.cpp built-in template by name

The value is passed to llama.cpp as the chat_template field in the request body. llama.cpp accepts either a built-in template name (e.g. llama3, qwen2.5, mistral) or a raw Jinja2 template string.

This is a client-side setting — different clients can use different templates for the same model name. Callers do not need to know or specify the template.

1 — Auth and body

The HTTP goroutine extracts the bearer key and validates it against the API key store (linear scan of state.json keys). The request body is read fully into memory with a hard cap of 1 MB.

2 — Inbound translation

The raw JSON is normalised into a format-agnostic InferenceRequest. Message content is stored as raw JSON (json.RawMessage), preserving tool_use/tool_result blocks, tool_calls arrays, and multi-part content without modification. Tools and tool_choice are forwarded as-is. The original wire format (openai / anthropic / openai-responses) is stored as SourceFmt for the outbound formatter.

n, logprobs, seedNot modelled; llama.cpp always returns one choice
Stop sequences, penaltiesNot forwarded to llama.cpp
Anthropic temperature / top_pNot parsed by AnthropicInbound
Responses API systemNot parsed by ResponsesInbound
3 — Model / alias validation

The requested model is checked against connected clients (model names) and the server-side alias map (from state.json). If the name is an alias, the target model must be served by a connected client. On failure the request is rejected immediately with HTTP 400 and a list of available models and aliases. The original model string (possibly an alias) is kept in the request for the scheduler.

4–5 — Queue and correlation

A UUID is assigned, priority is read from the API key, owner is set from the API key, and a buffered channel (cap 32) is created in the correlation store. The scheduler is woken via a coalescing signal channel (cap 1).

6–9 — Scheduler dispatch

Client-centric affinity scheduling: for each available client, PeekBestForClient finds the best queued request whose model matches the client’s model names or the server-side alias map, preferring the client owner’s requests. The globally best (client, request) pair wins. PopByID atomically commits the choice. If req.Model is an alias, it is rewritten to the canonical model name using the server-side alias map before the job is sent.

10–12 — Worker processing

The worker dispatches each JobMsg to a goroutine after acquiring a semaphore slot. llmesh-client: re-POSTs the full conversation history to llama.cpp — no cross-request KV cache management at the llmesh layer. llmesh-shim: dispatches to an external HTTP API (OpenAI or Anthropic format) or runs a shell command adapter. In both cases token deltas and tool-call deltas are forwarded back as ChunkMsg.

13 — Correlation and response

correlation.Send is non-blocking: if the 32-slot channel is full the chunk is silently dropped. The HTTP goroutine drains the channel and writes SSE events, flushing after each token or tool-call delta.
